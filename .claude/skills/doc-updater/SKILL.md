# 文档一致性维护助手

## 核心职责

**全量检查**项目文档之间的一致性，确保：
- 同一概念在各文档中描述一致
- 数量、版本、技术细节跨文档统一
- 功能范围、接口规格同步更新

## 全量检查清单

执行文档更新时，**逐项检查**以下内容：

### 1. 数量一致性

| 检查项 | 涉及文档 | 检查方法 |
|--------|---------|---------|
| ClickHouse 表数量 | 数据字典、项目索引（如有）、实施路线图、ETL 实施步骤等 | 以数据字典为准，与各文档对比，不写死具体数字 |
| Dashboard Tab 数量 | PRD、实施路线图、API 接口文档 | 确认与 PRD/路线图一致（如 8 个 Tab） |
| 预览页面数量 | 项目索引（如有）、实际 preview 目录 | `ls preview/` 与文档对比 |
| Agent 数量 | Agent 设计、技术选型 | Supervisor + 3 Specialist |
| 角色数量 | 权限矩阵、PRD | 4 级角色 |

### 2. 版本一致性

| 检查项 | 涉及文档 | 检查方法 |
|--------|---------|---------|
| 文档版本号 | 所有文档顶部 | 版本号是否反映最新修改 |
| 依赖版本 | 技术选型、实施路线图 | Vue/FastAPI/ClickHouse 版本一致 |
| 项目索引中的版本列表 | 项目索引（如有） | 若存在则检查是否列出最新文档版本 |

### 3. 命名一致性

| 检查项 | 涉及文档 | 检查方法 |
|--------|---------|---------|
| 字段命名 | 数据字典、API 接口文档 | 全部使用 snake_case |
| 表名 | 数据字典、实施路线图 | 表名完全一致 |
| ETL 脚本名 | 实施路线图、数据查询与Cube对接指南、ETL设计与数据流、Dev 游戏 Cube.js 查询手册 | 脚本/命令名与路线图及数据查询指南一致 |
| 接口路径 | API 接口文档、实施路线图 | `/api/v1/...` 格式一致 |

### 4. 功能与结构一致性

| 检查项 | 涉及文档 | 检查方法 |
|--------|---------|---------|
| 路线图第 10/11 节与数据字典及实施步骤 | 实施路线图、数据字典（路线图 §4.3.2 为 ETL 实施步骤，已并入） | 表、阶段、步骤描述一致 |
| 一期功能范围 | PRD、实施路线图、技术选型 | Dashboard/明细/告警/权限/ETL |
| 二期功能范围 | PRD、实施路线图、技术选型 | AI Agent 系统 |
| 已删除功能 | 全部文档 | 埋点功能已删除，不应出现 |

### 5. 接口一致性

| 检查项 | 涉及文档 | 检查方法 |
|--------|---------|---------|
| Dashboard 接口 | API 接口文档、实施路线图 | 8 个 Tab 接口对应 |
| 图表 ID | API 接口文档、前端实现 | charts 参数值一致 |
| SSE 流式接口 | API 接口文档、实施路线图 | `/api/v1/chat/stream` |

## 更新工作流程

### 步骤 1：全量读取

按文档清单所列文档，读取关键信息：
- 版本号
- 表/接口/功能数量（以数据字典等权威文档为准）
- 技术栈版本

### 步骤 2：交叉对比

按检查清单逐项对比，记录不一致项：

```
不一致项清单：
1. [数据字典] 表数量 N vs [实施路线图] 描述 M
2. [ETL 实施步骤] 脚本名 vs [数据查询与Cube对接指南] 不一致
3. [实施路线图] 相关文档表或专项计划索引与仓库实际文件不一致
4. ...
```

### 步骤 3：批量修复

1. 按发现顺序逐个修复
2. 更新版本号和修订记录
3. 保持格式一致

### 步骤 4：输出报告

```markdown
## 文档全量检查报告

### 检查范围
- 检查文档：文档清单所列（共 N 个）
- 检查项：全量检查清单所列

### 发现问题
| 序号 | 文档 | 问题 | 修复 |
|------|------|------|------|
| 1 | 实施路线图 | 表数量与数据字典不一致 | 已修复 |
| 2 | 实施路线图 §13 | 相关文档表与仓库实际路径一致 | 已修复 |

### 更新的文档
| 文档 | 版本变化 | 修改内容 |
|------|----------|----------|
| 实施路线图 | - | 表数量与数据字典对齐 |

### 无需修改
- PRD：功能范围一致
- 技术选型：版本一致
```

## 常见问题模式

### 模式 1：新增表后遗漏更新

**症状**：数据字典新增表，但项目索引（如有）、实施路线图等未更新
**检查**：以数据字典为准，对比其他文档中的表数量描述
**修复**：更新所有提及表数量的位置

### 模式 2：删除功能后遗漏清理

**症状**：PRD 删除功能（如埋点），但其他文档仍有残留
**检查**：全文搜索已删除功能的关键词
**修复**：删除所有相关描述

### 模式 3：接口变更后文档滞后

**症状**：API 接口文档更新了参数，实施路线图未同步
**检查**：对比 API 文档与实施路线图中的接口描述
**修复**：同步接口规格

### 模式 4：版本号未更新

**症状**：文档内容已修改但版本号未递增
**检查**：git diff 与版本表格对比
**修复**：递增版本号并添加修订记录

### 模式 5：专项计划已并入路线图但 plans 仍保留旧文件或索引未更新

**症状**：实施路线图已吸收某专项计划内容，但路线图 §13 相关文档表或正文中仍引用已删除或已合并的文档路径（如原 docs/plans/、docs/archive/、ai_对话栏修改计划、ETL-T3.5 独立文件、00-开发约定、06 下 00～06 单文件、页面实现检查报告.md）
**检查**：对比路线图 §13 相关文档表与仓库 docs 下实际文件；全文搜索上述已合并/已删文档路径
**修复**：将引用改为实施路线图对应节（如 §0、§4.3.2、§5.5.5、§12）或《页面规格说明》对应锚点；更新 §13 表与专项计划索引说明
