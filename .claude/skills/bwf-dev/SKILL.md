# /bwf-dev - 自动开发闭环

<command-name>bwf-dev</command-name>

## ⚠️ 关键执行要求（必须遵守）

**你必须在执行过程中持续输出进度，让用户知道正在发生什么！**

### 1. 启动时必须输出
```
═══════════════════════════════════════════════════════════════
  BW Framework 开发启动
═══════════════════════════════════════════════════════════════

📄 Plan: {plan_file}
📋 任务数: {total_tasks}
🔄 模式: {--skip-review ? "快速模式" : "完整模式"}

开始执行...
```

### 2. 每个任务开始时必须输出
```
───────────────────────────────────────────────────────────────
▶ Task {id}: {title}
───────────────────────────────────────────────────────────────
```

### 3. 每个任务完成时必须输出
```
✅ Task {id} 完成 (轮次: {rounds}, 审核: {review_status})
   Commit: {commit_hash}
```

### 4. 任务失败时必须输出
```
❌ Task {id} 失败 (轮次: {rounds}/{max_rounds})
   错误: {error_summary}
   需要人工处理
```

### 5. 全部完成后必须输出完整报告
（见下方"输出示例"章节）

### 6. 必须使用 TodoWrite 跟踪进度
在开始执行前，使用 TodoWrite 创建任务列表，并在执行过程中更新状态。

### 7. 必须写入进度文件
每完成一个任务，立即写入 `.bwf/progress.yaml`。

### 8. 必须更新 index.md 状态
当 Plan 的所有任务完成后，更新 `docs/plans/index.md` 中对应 Plan 的状态：
- `待审核` → `✅ 已完成`
- 如果有任务失败：`待审核` → `⚠️ 部分完成`

## 核心理念

> **AI 持续开发，直到工作完成**

本 Skill 实现自动化开发闭环：
1. 读取 Plan 中的任务
2. 逐个执行任务（写代码）
3. 自动验证（编译、测试、lint）
4. 代码审核（风格、架构）
5. 失败则自动修复
6. 通过则提交，继续下一个

## 语法

```bash
/bwf-dev <plan-file> [选项]

# 示例
/bwf-dev docs/plans/plan-admin.md           # 开发指定 Plan
/bwf-dev docs/plans/plan-admin.md --task=1.2  # 只执行指定任务
/bwf-dev --all                               # 按顺序开发所有 Plan
/bwf-dev --continue                          # 从上次中断处继续
/bwf-dev --max-fix-rounds=5                  # 每任务最多修复 5 轮
/bwf-dev --skip-review                       # 跳过代码审核（快速模式）
```

## 参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `<plan-file>` | Plan 文件路径 | 必填（除非 --all） |
| `--max-fix-rounds` | 每任务最大修复轮次 | 3 |
| `--task` | 只执行指定任务 | 执行全部 |
| `--continue` | 从上次中断处继续 | false |
| `--all` | 按依赖顺序开发所有 Plan | false |
| `--skip-review` | 跳过代码审核 | false |

## --all 模式执行逻辑

使用 `--all` 参数时，必须读取 `docs/plans/index.md` 中的**执行批次**表格。

### 默认行为：串行执行

**默认情况下，所有 Plan 串行执行**，原因：

| 问题 | 说明 |
|------|------|
| 文件冲突 | 多个 Plan 可能同时修改 package.json、tsconfig.json 等共享文件 |
| Git 冲突 | 并行 commit 可能产生 merge 冲突 |
| 上下文爆炸 | 并行运行多个长任务会超出 token 限制 |
| 调试困难 | 并行执行出错时难以定位问题 |

### 执行规则
1. **读取 index.md** 中的"执行批次"章节
2. **跳过已完成的 Plan**（状态为 "✅ 已完成"）
3. **按批次顺序执行**：
   - 同一批次内的 Plan 按表格顺序**串行**执行
   - 完成当前批次所有 Plan 后，才能进入下一批次
4. **检查依赖**：执行 Plan 前确认其依赖的 Plan 已完成

### 执行流程
```
1. 读取 docs/plans/index.md
2. 解析"执行批次"表格
3. 解析"Plan 列表"表格的状态列
4. 过滤出未完成的 Plan
5. 按批次顺序执行：
   for each batch:
     for each plan in batch:
       if plan.status != "✅ 已完成":
         执行 /bwf-dev {plan_file}
```

### 并行执行（实验性）

如果确实需要并行执行同一批次内的 Plan，可以：

```bash
# 手动在多个终端并行执行不同 Plan
# 终端 1:
/bwf-dev docs/plans/plan-user-system.md

# 终端 2:
/bwf-dev docs/plans/plan-sdk-core.md

# 终端 3:
/bwf-dev docs/plans/plan-sdk-android.md
```

**注意**：并行执行时需自行处理文件冲突和 Git 冲突。

## 两层验证机制

### 硬验证（auto-developer 执行）

机器可验证的标准，**必须通过**：

| 类型 | 命令 | 通过条件 |
|------|------|----------|
| 编译 | `pnpm run build` | exit_code == 0 |
| Lint | `pnpm run lint` | exit_code == 0 |
| 测试 | `pnpm run test` | exit_code == 0 |
| 自定义 | Plan 中定义 | 按定义判断 |

### 软审核（code-reviewer 执行）

机器**无法验证**的标准，可选：

| 维度 | 检查内容 |
|------|----------|
| 命名规范 | 变量/函数命名是否清晰 |
| 代码结构 | 函数长度、文件大小、重复代码 |
| 架构合规 | 分层、依赖方向、设计模式 |
| 可读性 | 注释、魔法数字、嵌套层级 |
| 潜在问题 | 性能、安全、资源泄漏 |

---

## 安全/边界快速检查（条件触发）

**触发条件**：涉及认证/凭证、文件上传下载、缓存/软删除、响应头/重定向、队列/后台任务。

**必须逐条确认**：
- 凭证变更/软删除后**立即**失效（缓存清理、旧 key 失效）。
- 上传/下载**有大小限制**且错误码明确。
- 响应头拼接**无注入风险**（文件名/URL 需要清洗或安全生成）。
- 记录日志时**不泄露密钥/令牌**。
- 入口有**速率/滥用**保护或明确不需要。

## 验证命令规范

Plan 中的每个 Task 必须定义验证命令：

```markdown
### Task 1.2: 实现用户认证 API

**验证**:
- `pnpm run build` → exit_code == 0
- `pnpm run test -- --grep "auth"` → exit_code == 0
- `curl localhost:17060/api/health` → contains "ok"
```

**验证类型**:

| 类型 | 语法 | 示例 |
|------|------|------|
| 退出码 | `→ exit_code == 0` | `pnpm run build → exit_code == 0` |
| 包含文本 | `→ contains "text"` | `curl /api → contains "ok"` |
| 文件存在 | `→ file_exists` | `ls dist/index.js → file_exists` |
| 正则匹配 | `→ matches /regex/` | `cat log → matches /success/i` |

---

## 自修复机制

当验证失败时，auto-developer 自动：

1. **解析错误信息**
   ```
   error TS2322: Type 'string' is not assignable to type 'number'.
     src/services/auth.ts:45:10
   ```

2. **定位问题**
   - 提取文件路径: `src/services/auth.ts`
   - 提取行号: `45`
   - 提取错误类型: `TS2322`

3. **读取上下文**
   - 读取问题文件
   - 读取相关类型定义
   - 读取 Plan 中的预期行为

4. **生成修复**
   - 修改代码
   - 重新运行验证
   - 记录修复历史

5. **退出条件**
   - ✅ 验证全部通过 → 进入代码审核
   - ❌ 达到 max_fix_rounds → 标记需人工处理

## 代码审核流程

在硬验证通过后，调用 code-reviewer agent：

```yaml
Task:
  subagent_type: code-reviewer
  prompt: |
    ## 审核任务

    Task ID: {task_id}
    Task 描述: {task_description}

    ## 变更文件

    {files_changed}

    ## 项目规范

    {project_conventions}

    ## 要求

    审核代码的：
    1. 命名规范
    2. 代码结构
    3. 架构合规
    4. 可读性
    5. 潜在问题
```

**审核结果处理**:

| 结果 | 处理方式 |
|------|----------|
| `approved` | 直接提交 |
| `approved_with_suggestions` | 提交，记录建议到报告 |
| `needs_changes` | auto-developer 修复，round++，重新审核 |

## 进度持久化（强制）

**必须**在每个任务完成后立即更新进度文件。

### 文件结构（按 Plan 分文件）

```
.bwf/
└── progress/
    ├── plan-infrastructure.yaml
    ├── plan-user-system.yaml
    ├── plan-sdk-core.yaml
    └── ...
```

**文件命名规则**: 取 Plan 文件名（不含 `.md`），如 `docs/plans/plan-admin.md` → `.bwf/progress/plan-admin.yaml`

### 写入时机
1. **启动时**: 创建/更新 `.bwf/progress/{plan-name}.yaml`，标记所有任务为 `pending`
2. **任务开始时**: 更新当前任务状态为 `in_progress`
3. **任务完成时**: 更新任务状态为 `completed`，记录 commit hash
4. **任务失败时**: 更新任务状态为 `failed`，记录错误信息

### 文件格式
```yaml
# .bwf/progress/plan-admin.yaml
plan: docs/plans/plan-admin.md
started_at: 2026-02-01T10:00:00Z
updated_at: 2026-02-01T10:30:00Z
status: in_progress  # pending | in_progress | completed | failed
tasks:
  - id: "1.1"
    status: completed
    rounds: 1
    review: approved
    commit: "abc1234"
  - id: "1.2"
    status: completed
    rounds: 2
    review: approved_with_suggestions
    suggestions:
      - "建议将魔法数字提取为常量"
    commit: "def5678"
  - id: "1.3"
    status: in_progress
    rounds: 0
  - id: "1.4"
    status: pending
```

### 写入代码示例
```bash
# 使用 Write 工具写入进度文件
mkdir -p .bwf/progress
# 然后使用 Write 工具写入 .bwf/progress/{plan-name}.yaml
```

### 并行执行支持

由于每个 Plan 的进度独立存储，可以在多个终端并行执行不同 Plan：

```bash
# 终端 1
/bwf-dev docs/plans/plan-user-system.md

# 终端 2
/bwf-dev docs/plans/plan-sdk-core.md
```

进度文件不会冲突。

使用 `--continue` 可从中断处恢复，会自动读取对应 Plan 的进度文件。

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| Plan 文件不存在 | 报错，提示先 `/bwf-plan` |
| 任务依赖未满足 | 先执行依赖任务 |
| 硬验证超过 max_fix_rounds | 暂停，标记需人工处理 |
| 代码审核超过 max_fix_rounds | 暂停，标记需人工处理 |
| 网络/环境错误 | 重试 3 次，失败则暂停 |

---

## 安全机制

1. **每任务提交**: 每个任务完成后立即 git commit，失败可回滚
2. **进度持久化**: 中断后可恢复
3. **人工检查点**: 超过修复轮次时暂停等待人工决策
4. **两层验证**: 硬验证 + 软审核双重保障
5. **审核建议记录**: 即使通过也记录优化建议

## 快速模式

使用 `--skip-review` 跳过代码审核，只做硬验证：

```bash
/bwf-dev docs/plans/plan-admin.md --skip-review
```

适用场景：
- 快速原型开发
- 已有完善的 lint 规则
- 时间紧迫的修复

## 执行检查清单（每次执行必须核对）

在执行 /bwf-dev 时，**必须**按以下步骤执行：

### 启动阶段
- [ ] 读取 Plan 文件
- [ ] 解析所有任务和依赖关系
- [ ] 输出启动信息（Plan 名称、任务数、模式）
- [ ] 使用 TodoWrite 创建任务列表
- [ ] 创建 `.bwf/progress/{plan-name}.yaml` 文件

### 每个任务
- [ ] 输出任务开始信息
- [ ] 更新 `progress/{plan-name}.yaml` 状态为 in_progress
- [ ] 更新 TodoWrite 状态为 in_progress
- [ ] 执行开发（写代码）
- [ ] 运行验证命令
- [ ] 如果失败：分析错误、修复、重试（最多 max_fix_rounds 轮）
- [ ] 如果通过：运行代码审核（除非 --skip-review）
- [ ] 如触发条件满足：执行“安全/边界快速检查”
- [ ] Git commit
- [ ] 输出任务完成信息
- [ ] 更新 `progress/{plan-name}.yaml` 状态为 completed
